OAuth 2.0 — протокол делегированной авторизации

OAuth 2.0 — это отраслевой стандартный протокол для авторизации, который позволяет приложениям получать ограниченный доступ к учетным записям пользователей на HTTP-сервисах.

Видеообзор OAuth 2.0

Роли в OAuth 2.0

Resource Owner

Пользователь, который предоставляет доступ к своим ресурсам.

Client

Приложение, запрашивающее доступ от имени пользователя.

Authorization Server

Аутентифицирует пользователя и выдает токены.

Resource Server

Размещает защищенные ресурсы и принимает access token.

Потоки авторизации

Authorization Code + PKCE

  • Redirect пользователя
  • Аутентификация
  • Получение кода
  • Обмен кода на токен

Client Credentials

Используется для сервер-серверной аутентификации.

Implicit Flow

Использовался для SPA, сейчас не рекомендуется.

Resource Owner Password

Только для доверенных клиентов.

Пример реализации OAuth

// Формирование URL авторизации
const authUrl = new URL('https://authorization-server.com/authorize');
authUrl.searchParams.set('response_type', 'code');
authUrl.searchParams.set('client_id', CLIENT_ID);
authUrl.searchParams.set('redirect_uri', REDIRECT_URI);

window.location.href = authUrl.toString();

Потоки авторизации (Grant Types)

Authorization Code Flow

Использование: Веб и мобильные приложения

Описание: Самый безопасный поток с получением кода авторизации.

Шаги:
  1. Redirect пользователя
  2. Аутентификация и согласие
  3. Получение authorization code
  4. Обмен кода на access token

Implicit Flow

Использование: Старые SPA

Описание: Токен возвращается напрямую.

Не рекомендуется для новых приложений.

Resource Owner Password

Использование: Доверенные клиенты

Описание: Пользователь передает логин и пароль приложению.

Использовать только в контролируемой среде.

Client Credentials

Использование: Сервер-сервер

Описание: Аутентификация без пользователя.

Authorization Code + PKCE

Использование: Мобильные и SPA

Описание: Улучшенная версия Authorization Code.

PKCE добавляет защиту от перехвата кода.

Токены в OAuth 2.0

Access Token

  • Короткое время жизни
  • Передается в Authorization: Bearer
  • Может быть JWT

Refresh Token

  • Долгоживущий
  • Используется для обновления access token
  • Может быть отозван

ID Token

  • Содержит данные пользователя
  • Используется для аутентификации

Реализация OAuth 2.0

Регистрация приложения

{
  "client_id": "...",
  "redirect_uri": "https://site.com/callback",
  "scope": "read profile email"
}

Инициирование авторизации

authUrl.searchParams.set('response_type', 'code');
authUrl.searchParams.set('client_id', CLIENT_ID);
authUrl.searchParams.set('redirect_uri', REDIRECT_URI);

Использование Access Token

fetch('/api/user', {
  headers: {
    'Authorization': `Bearer ${accessToken}`
  }
});

Безопасность OAuth 2.0

CSRF Protection

Используйте параметр state.

PKCE

Обязательно для мобильных и SPA.

redirect_uri validation

Строго проверяйте соответствие зарегистрированному URI.

Минимальный scope

Запрашивайте только необходимый доступ.

HTTPS

Всегда используйте защищенное соединение.
